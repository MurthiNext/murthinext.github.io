近日，我在尝试使用以下命令激活我的 Windows 11 系统，然而出了点岔子。
`irm https://get.activated.win | iex`
等待了很长时间之后，依然没有成功激活，我又试了下手动激活，同样的，无法连接到 KMS 服务器。
这让我想起了之前连接 `ssh.github.com` 时遇到了同样的问题，让 DeepSeek V4 Pro 帮我探查一番，果然是又出了问题。
简单来说，我的代理服务挂在局域网的 192.168.0.169 主机下的 7890 端口上（使用的  Docker 版的 Mihomo），并将 DNS 指向了 192.168.0.169，开启了 Fake-IP，一般而言这样是没有问题的，然而总会有些意外。
如果一个服务绕过了系统代理，直接请求连接的话，他的 DNS 请求会根据我的系统设置，照常发送给我的代理服务器——Mihomo 见到这个请求，直接甩回来了一个 Fake-IP，这时候问题就来了，由于这个流量没有经过系统代理，他相当于直接尝试连接 198.18.xx.xx 这个地址，很显然是不可能的，所以会导致 Connection failed。
解决问题很简单，就是把系统默认 DNS 设置成普通的 DNS 服务器，比如 223.5.5.5，这样在流量不走系统代理时，也可以正常访问普通 DNS 服务器，直接连接远程主机——虽然没有经过代理，但至少不会连接失败。
总而言之就是，别闲得没事去改默认 DNS，要么让代理服务器解析域名，要么 TUN 全部转发，手动指向一个不那么正常的 DNS 服务器就会导致这种尴尬情况。。。
